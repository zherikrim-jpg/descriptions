# Обновление OTS и GRP
## Обновление через Admetrix
Admetrix - сервис с данными по рекламным конструкциям во всех крупных городах РФ.
У каждой РК в Admetrix есть множество полей: тип, формат, город, владелец, координаты, месяц, кто размещался, за какую сумму и т.д.
Для обновления нас интересует 5 полей: id_geo (уникальный id), gid, abc (направление РС), grp и ots. Примеры полей можно посмотреть в таблице analytics.admetrix_data.
Порядок обновления:
1. id_geo указан на наших РС/РП в нашей, это поле code_espar в sides в случае digital и так же code_espar в surfaces в случае, если тип != digital.
Нужно сделать выгрузку данных из Admetrix через [скрипт](https://github.com/zherikrim-jpg/python/blob/main/parser_adm.py), загрузить данные в analytics.admetrix_data и далее сджойнить по code_espar grp и ots.
Может быть такое, что md_grp и md_ots будет указан в sides и surfaces, а при джойне видим, что там null, тогда оставляем старые значения и в файл импорта не добавляем. Если сджойненные grp и ots не null, то подставляем новые.
После того как сджойнили и получили таблицу формата Город, Тип смены изображения, Уникальный ID, GRP, OTS, показывать Admetrix, нужно сделать импорт этой таблицы в ERP.
Поля в файле импорта:
- Город, это ясно
- Тип смены изображения: digital, статика и т.д.
- Уникальный ID: rim_gid из sides для digital и rim_gid из surfaces для != не digital
- GRP: grp из Admetrix
- OTS: ots из Admetrix
- Показывать Admetrix: значения да/нет. В данном случае будет "да"
2. id_geo (code_espar) не указан на наших РС/РП
  В таком случае нужно сначала найти id_geo из Admetrix для наших РС/РП. Это можно сделать через джойн по полям city-gid-abc из Admetrix, где seller = РИМ MEDIAGROUP. Справа будут поля из наших таблиц city-gid-direction. Получится, что сначала для digital нужно собрать таблицу из sides, places, cities, structures и для surfaces: surfaces, sides, places, cities, structures. Либо использовать данные из analytics.sides_counting.

ВАЖНО! В Admetrix digital стороны имеют поверхности, соотвественно на один gid приходится, скажем, 10 id_geo. Когда будете джойнить как в п.2, то у вас  разможатся строки на кол-во поверхностей, которое есть для этого gid из наших таблиц, чтобы этого избежать можно группировать данные Admetrix по city, gid, direction, grp, ots и агрегировать id_geo через strng_agg() и брать только первое значение из строки, так можно делать, потому что grp и ots одинаковый на каждой поверхности для одного гида. То есть нам не сильно важно, какой именно id_geo из 10 будет стоять у нас в code_espar.

Далее после джойнов, так же как и в п.1, составляется файл импорта, там где id_gep нашелся и grp и ots не пусты, добавляем в файл и импортируем.

3. Если после всех манипуляций id_geo найти не удалось, либо удалось, но grp и ots пусты, то нужно заполнить grp и ots через [catboost_ots.py](https://github.com/zherikrim-jpg/python/blob/main/catboost_ots.py). Суть в чем: модель обучается на данных Admetrix и проставляет grp и ots на наших РС/РП. Таблица с данными для обучения: analytics.admetrix_data и таблица с данными для заполнения analytocs.sides_for_calc. Для наших городов, которых в Admetrix нет, для заполнения grp и ots всегда используется эта модель. Результаты заполнения скрипт выгружает в [таблицу](https://docs.google.com/spreadsheets/d/1O0ZIMwlh8bT5WEEIwPh41hj5pVY6n6dkNuMUcPyEqjY/edit?gid=163391055#gid=163391055) на лист Заполненные значения и на лист Файл импорта. Важно, чтобы поля population_18+ и еще несколько не были пустыми, для этого ведется справочник Список городов, где по мере появления новых городов нужно дополнять этот лист и импортировать в postgres через [скрипт](https://github.com/zherikrim-jpg/python/blob/main/cu_cities_ml.py) . Данные в справочнике: subject, city, population, population_18+, strategy, latitide_cent и longitude_cent берутся из отрытых источников (кроме strategy). Перед тем как делать файл импорта, нужно вручную проверить, что насчитала модель. Негласные правила такие:
- ots у А стороны выше, чем у B у одного и того же гида
- grp = 100 * ots * 1000 / population_18+. По логике, он не может быть > 100. И в целом, значения от 50 до 100 будут выглядеть подозрительными. Пример: город N с населением 10_000. Модель дала ots для гида в этом городе 20 (напоминаю, что ots измеряется в тысячах). То есть потенциальных неуникальных просмотров 20_000, что выше в 2 раза, яем население в городе => grp = 200. В таком случае нужно в ручную занизить ots до 2-3 и пересчитать grp.
- для медиафасадов (не медиаскринов) ots всегда завышается вручную. Для миллионников это всегда около 150, для городов с меньшим населением какой ots дать можете посмотреть сами в analytics.sides_counting с show_admetrix_stats is false и structure_category = Медиафасад.
Нужно избавиться от ручного способа корректировки данных, которая дает модель, но как я не придумал.
После того как все проверено, создается файл импорта с полями Город, Тип смены изображения, Уникальный ID, Рассчитываемые GRP, Рассчитываемые OTS, показывать Admetrix. Все то же самое, как в п.1, только показывать Admetrix = нет и названия полей, который содержат данные grp и ots будут отличаться, на этом все.

Импорт производится так: скачивается гугл таблица с данными по grp и ots в .xlsx формате и импортируется через админку ERP по пути: Импорт/Экспорт - Импорт инвентаря (обновление) - Добавить inventory update import - Browse - выбираете файл - сохранить и выйти, ждете. Если никаких ошибок не возникло при импорте, то все окей, если ошибки появились (такое редко бывает) и самостоятельно разобраться не получилось через скачивание ошибок, то пишите Артему Дащенко, он поможет разобраться.
